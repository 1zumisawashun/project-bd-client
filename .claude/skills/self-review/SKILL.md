# Self Review

開発者が好きなタイミングで呼び出すセルフレビュースキル。
現在のdiffに対してフィードバックを提供し、レビュー内容から役立ちそうなスキルがあれば提案する。

## When to Use This Skill

- 実装の区切りでフィードバックが欲しいとき
- コミット前にdiffを見直したいとき
- 「レビューして」「F/Bちょうだい」と言われたとき

## ワークフロー

### Step 1: diff の収集

現在の変更差分を収集する。

```bash
# 未コミットの変更（ステージ済み + 未ステージ）
git diff HEAD

# 変更がなければ直近コミットのdiffを使う
git log --oneline -5
git diff HEAD~1..HEAD
```

- 未コミットの変更がある場合はそちらを優先
- 変更がなければ直近コミットのdiffをレビュー対象にする
- diffの規模が大きい場合はユーザーに確認してスコープを絞る

### Step 2: セルフレビュー（メイン）

diffを分析し、実用的なフィードバックを返す。

#### レビュー観点

1. **コード品質**: 重複、不要なコード、命名の改善余地
2. **セキュリティ**: OWASP Top 10に該当するリスク（XSS, インジェクション等）
3. **パフォーマンス**: 不要な再レンダリング、N+1クエリ、重い処理
4. **テスト**: 変更に対してテストが十分か、カバレッジの漏れ
5. **型安全性**: any の使用、型の緩さ、as によるキャスト

#### 出力フォーマット

```markdown
## Self Review

### 変更サマリー

- 変更ファイル数: X
- 主な変更内容: （簡潔に）

### フィードバック

（実際に問題になりうるもののみ。問題なければ「特に指摘事項はありません」）

- **[カテゴリ]** `ファイルパス:行番号` - 指摘内容と改善案

### Good

（良いプラクティスがあれば1-2個ピックアップ）
```

#### レビューの姿勢

- **過度な指摘は避ける**: 実際に問題になりうるもののみ。「念のため」レベルの指摘は不要
- **改善案をセットで出す**: 問題点だけでなく「こうすると良い」まで提示する
- **ポジティブなフィードバックも含める**: 良い書き方があれば素直に褒める
- **修正が必要な場合は修正するか確認する**: 大きな問題を見つけたらスキル提案より先に修正を提案する

### Step 3: おすすめスキルの提案（サブ）

Step 2のレビュー結果とdiffの内容から、役立ちそうなスキルを提案する。

#### 分析対象

- diffで触れた技術（フレームワーク、ライブラリ）
- 作業の種類（新機能、バグ修正、リファクタリング、テスト追加）
- レビューで見つかったパターン（フォーム処理、API連携、状態管理等）

#### スキル検索

```bash
npx skills find [keyword]
```

キーワード例:

- Reactコンポーネントを作った → `react`, `accessibility`
- APIルートを追加した → `api`, `security`
- テストを書いた → `testing`
- UI/CSSを変更した → `design`, `css`
- DB関連の変更 → `database`

#### 出力フォーマット

```markdown
### おすすめスキル

（レビュー内容から関連しそうなスキルがあれば）

- **スキル名**（インストール数） - なぜおすすめか
  `npx skills add owner/repo@skill-name`
```

#### 提案ルール

- 提案は **1-3個** に絞る
- インストール数が少ない（100未満）スキルは推奨しない
- 既にインストール済みのスキルは除外（`.claude/skills/` を確認）
- 関連スキルが見つからなければ無理に提案しない（「特におすすめはありません」でOK）
- スキル提案はあくまでおまけ。メインはレビューのフィードバック
